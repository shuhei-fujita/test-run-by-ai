## スコープ

```
テスト計画 → 設計 → 実行 → 評価
```

| フェーズ | 内容                               | 自動化対象 |
| -------- | ---------------------------------- | ---------- |
| 計画     | 実行順序・依存関係の整理           | 対象       |
| 設計     | テスト観点の洗い出し・シナリオ作成 | 対象外     |
| 実行     | ブラウザ操作・目視確認・スクショ   | 対象       |
| 評価     | 結果記録・進捗管理・報告           | 対象       |

設計（テスト観点の洗い出し・シナリオ作成）は属人性が高く、対象外。

## PoC 実績サマリ

### 実行規模

- 対象: 3画面・170+シナリオ
- **実行可能シナリオの PASS 率: 92%**
- BLOCKED の主因はテストデータ不足（テスト環境の準備不足）であり、ツール側の制約ではない

### FAIL の内訳

| 原因分類               | 件数 | 備考                               |
| ---------------------- | ---- | ---------------------------------- |
| プロダクトバグの発見   | 2件  | UI実装漏れを検出                   |
| テストスイートとの乖離 | 1件  | テスト設計と実装の差異。仕様確認要 |
| 環境問題               | 1件  | APIサーバーエラー                  |

FAIL 4件中 2件がプロダクトバグの発見 = **テストとして正しく機能している**。

### HITL 介入実績

| 介入理由               | 発生回数 | 対応                                   |
| ---------------------- | -------- | -------------------------------------- |
| テストデータ不足       | 3回      | 必要データを提案→人間が準備→テスト再開 |
| 環境問題（API エラー） | 1回      | サーバー側問題として記録、テスト続行   |
| テストスイートとの乖離 | 1回      | 実装差異を記録、仕様確認要として保留   |
| BLOCKED 蓄積           | 1回      | ブロッカー多数→方針転換を相談          |

HITL 設計が想定通りに機能。AIが暴走せず適切なタイミングで停止している。

## フェーズ別の削減率

| フェーズ | 従来（手動）                           | 導入後                                     | 削減率（実感） |
| -------- | -------------------------------------- | ------------------------------------------ | -------------- |
| 計画     | 手動で実行順序・依存関係を整理         | `/plan-test` で自動生成                    | 80〜90%        |
| 設計     | 観点洗い出し・シナリオ作成             | 人間が実施（対象外）                       | -              |
| 実行     | ブラウザ手動操作→目視確認→スクショ撮影 | `/playwright-cli` でAI操作・判定・スクショ | 60〜80%        |
| 評価     | Excelにスクショ貼付、手動集計・報告    | `task.md` に自動記録・進捗更新             | 90%以上        |

- 計画: task.md の自動生成は数分。依存関係・実行順序・事前準備チェックリストまで自動で整理される
- 評価: PASS/FAIL ステータス、スクリーンショット、備考がテスト実行と同時に task.md へ自動記録。Excel 転記ゼロ

## PoC で判明した課題と対策

| 課題                       | 影響                             | 対策                                             |
| -------------------------- | -------------------------------- | ------------------------------------------------ |
| テストデータ不足で BLOCKED | 実行可能率が画面により大きく変動 | テストデータ準備手順を事前に整備する             |
| bot検出（外部サイト）      | 外部サイトで発生しうる           | 自社アプリでは問題なし。外部サイトは対象外とする |
| セッションクラッシュ       | run-code + route の組み合わせ    | CDP throttling を使用。Tips に記載済み           |
| API従量課金の実コスト      | 未計測                           | 次回テスト実行時に Usage Dashboard で計測する    |
